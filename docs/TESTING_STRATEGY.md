# Стратегия тестирования

Контекст проекта: недельные спринты · 1 QA (ручник+автоматизатор) · 5 разработчиков ·
выкатка каждой проверенной задачи в прод · десктоп + мобильный веб (приоритет) ·
мобильное приложение (вторично) · два стенда для тестирования · автотесты пока
гоняются вручную · есть Playwright (`@critical`, visual, `prodSmoke`) и API-хелперы.

## 1. Главный вызов и принцип

При соотношении **1 QA на 5 разработчиков** + выкатке каждой задачи в прод QA физически
не может быть единственным гейтом качества. Базовый принцип — **shift-left: качество это
ответственность всей команды**; QA владеет *стратегией, рисками и автоматизацией*, а не
«прокликиванием всего руками».

Три опоры:
1. **Вовлечь разработчиков** в тестирование (unit/API + прогон `@critical` перед PR).
2. **Автоматический гейт в CI** вместо ручного прогона — главный текущий пробел.
3. **Risk-based приоритизация** — ограниченный ресурс направляем на критичные пути
   (auth, оплата, аплоад, плеер) в первую очередь.

Цель — превратить единственного QA из «бутылочного горлышка» в «мультипликатора».

## 2. Распределение ответственности (снимает bottleneck)

| Кто | Зона ответственности |
|---|---|
| **Разработчики** | unit-тесты на свою логику; API/integration на свои эндпоинты; прогон `@critical` локально перед PR; пишут/дополняют E2E для своей фичи на готовых Page Objects/паттернах |
| **QA** | владелец стратегии и risk-анализа; критичные E2E; **exploratory-тестирование новых фич**; поддержка фреймворка; релиз-гейт; ревью тест-кейсов разработчиков |

QA перестаёт быть «руками на каждую задачу» и становится архитектором качества +
последней exploratory-проверкой.

## 3. Пирамида тестирования под текущий стек

```
        ▲  E2E (Playwright, QA)        — критичные journey, desktop + mobile web. Тонкий слой.
       ███ Component / Visual          — уже есть visual-проекты (Docker)
      █████ API / Integration (dev+QA) — быстрее E2E; уже есть AuthApi/VideoApi/DatabaseHelper
     ███████ Unit (devs)               — бизнес-логика, дёшево и быстро
```

Правило: **дешёвое и быстрое — внизу и много; дорогое E2E — наверху и мало**, только на
критичные пути. Мобильное приложение (не приоритет) — **ручной smoke-чек-лист на
релизах**, автоматизация позже.

## 4. Процесс в недельном спринте

- **Планирование:** QA проставляет в задачах *критерии приёмки* и *risk-тег*
  (high/med/low) → задаёт глубину проверки.
- **Разработка:** dev пишет код + тесты, гоняет `@critical` локально перед PR.
- **Проверка задачи:** авто-регресс/`@critical` в CI на PR (блокирует мердж) + QA делает
  **exploratory** новой фичи на **Стенде A**.
- **Перед прод:** **smoke на Стенде B (pre-prod)** — блокирующий гейт.
- **После прод:** `prodSmoke` как пост-деплой мониторинг.

## 5. CI/CD — закрыть главный пробел

Сейчас тесты гоняются вручную. Минимальный пайплайн по росту зрелости:

1. **На PR:** lint + unit + API + `@critical` → **блокирует мердж**. Самый высокий ROI,
   делать первым.
2. **Перед деплоем в прод:** `@critical` + smoke на Стенде B → **блокирует выкатку**.
3. **После деплоя:** `prodSmoke` → алерт при падении.
4. **Ночью:** полный функциональный регресс + visual (Docker) на стенде.

Это одновременно разгружает QA и ставит страховку перед прод.

## 6. Два стенда — как развести

- **Стенд A (dev/staging):** тестирование фич, ветки, нестабильный — exploratory и прогон
  фич-веток.
- **Стенд B (pre-prod/release):** стабильный, прод-подобные данные — smoke/регресс-гейт
  перед выкаткой.

## 7. Устойчивость к частой смене вёрстки

Частая смена вёрстки бьёт по регрессу **двумя разными** способами — их нельзя путать,
лечатся они по-разному.

| Тип теста | Что ломается | Чем лечить |
|---|---|---|
| Функциональные E2E | локаторы (CSS-классы, структура DOM) | стабильные семантические якоря |
| Визуальные тесты | пиксели (любой сдвиг/цвет/отступ) | маски + узкий скоуп + не блокировать |

Функциональный тест **не должен падать от смены вёрстки вообще** — он проверяет
*поведение*, а не пиксели. Падает → проблема в локаторах.

### E2E: устойчивые локаторы — главный рычаг
Корень хрупкости — привязка к сгенерированным CSS-классам и структуре (в коде встречается
`mui-ub9qya`, `MuiBox-root mui-1laxava` — такие классы меняются при каждом
рестайлинге/обновлении MUI; это антипаттерн). Приоритет локаторов:
1. `data-testid` / `data-id` — контракт с разработчиками, не зависит от вёрстки (уже есть:
   `aitv-channel`, `aitv-share`, `aitv-short-card`).
2. `getByRole` / `getByLabel` — семантика.
3. `getByText` — для стабильного текста.
4. ❌ CSS-классы (особенно `mui-*`), глубокий XPath, `nth-child` — запрещены.

Это и **организационная мера**: `data-testid` у каждого интерактивного/проверяемого
элемента — часть Definition of Done; его не удаляют/не переименовывают без согласования.
Для одного QA на 5 devs это самый дешёвый способ убить хрупкость — «якоря» поддерживают
разработчики. (См. `docs/LOCATORS_POLICY.md`, если заведён.)

### Визуальные тесты: принять природу
Visual regression падает от любого пикселя по своей сути. При частой смене вёрстки нельзя
делать его широким блокирующим гейтом. Правила:
- **Не блокировать им мердж** — visual ночью / по запросу, с ручным апрувом диффа;
  блокирует только функциональный `@critical` (на testid, устойчив).
- **Узкий скоуп** — снимать критичные по дизайну зоны/компоненты, не страницу целиком.
- **Маскировать всё динамичное и часто меняющееся** (аватары, имена, счётчики — уже
  делается; расширять маски на нестабильные блоки).
- **Обновление эталонов — часть того же PR**, где меняют вёрстку (`--update-snapshots`,
  дифф ревьюится автором изменения, а не QA постфактум).
- Опционально — AI-визуал (Percy/Chromatic/Applitools): «осмысленное» сравнение, меньше
  шума, review-воркфлоу диффов; но это доп. сервис и деньги.

### Процесс
- Редизайн анонсируется заранее → QA готовит обновление масок/эталонов в том же спринте.
- `data-testid` — в DoD новых фич.
- В CI: `@critical` (functional, testid) блокирует; `visual` — ночью/информативно.

**Вывод:** инвестировать в стабильные локаторы, а не в количество visual-тестов.
Функциональный регресс на `data-testid` переживает рестайлинг почти без поломок; визуал
держать узким, маскированным, ночным и не-блокирующим.

## 8. Документация (вести в репозитории, markdown)

Держать рядом с кодом (ревьюится в PR, не устаревает как тяжёлый Confluence):

- `docs/features/<feature>.md` — поведение фичи, критичные сценарии, ссылки на
  автотесты/кейсы. Заводится вместе с фичей.
- `docs/checklists/smoke.md` и `docs/checklists/regress.md` — списки smoke (перед
  релизом) и регресс-проверок, с разбивкой **desktop web / mobile web / приложение**.
- `TEST_COVERAGE.md` — карта «что авто / ручное / TODO / blocked» (уже ведётся).
- Шаблоны bug-report и Definition of Done — короткие, в `docs/`.

Принцип: **лёгкая живая документация**, а не тяжёлый процесс.

## 9. Метрики прогресса

- Escaped defects (баги в проде) — тренд.
- % критичных путей под автотестами.
- Время прогона регресса.
- Доля задач с тестами от разработчиков.

## 10. Дорожная карта (по приоритету ROI для одного QA)

1. **Сейчас:** CI-гейт `@critical` + visual на PR (убрать ручной прогон) + smoke-чек-лист
   + DoD в задачах.
2. **Дальше:** вовлечь разработчиков (unit/API + прогон `@critical` перед PR); завести
   `docs/features` и чек-листы.
3. **Потом:** расширить E2E на критичные пути desktop + mobile web; ночной регресс;
   prod-smoke мониторинг.
4. **Позже:** мобильное приложение — ручной smoke-чек-лист, автоматизация по мере ресурса.
