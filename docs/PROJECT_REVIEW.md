# Архитектурное ревью проекта автотестов

Отчёт сеньор-ревьюера по проекту `aitv-autotests` (Playwright + TypeScript). Ниже — находки по убыванию критичности с конкретными путями и `file:line`. По каждой: **что не так → чем грозит → что делать**. Часть находок уже исправлена в рамках этого же ревью — они помечены **[ИСПРАВЛЕНО]**.

Дата: 2026-07-08.

---

## 1. Критичное

### 1.1. Документация лгала про `.env`-файлы **[ИСПРАВЛЕНО]**
- **Что не так.** `README.md:21` утверждал «не коммитятся, все в `.gitignore`», `CLAUDE.md:81` — «all gitignored». При этом `.env.web3tv`, `.env.web3tv2`, `.env.prod` реально закоммичены и отслеживаются git (`git ls-files` их показывает). Это осознанное решение: секретных данных в файлах нет, коммит упрощает онбординг и CI. Более того, `README.md:217` и `README.md:238` уже честно описывали это («env берётся из закоммиченных `.env.web3tv2`», «Env-файлы … уже в репозитории») — то есть документация противоречила сама себе.
- **Чем грозило.** Не проблема безопасности (секретов нет — это подтверждённое решение), а противоречие внутри документации: новый участник, прочитав line 21, ищет несуществующие файлы или пытается их создать, хотя они уже в checkout.
- **Что сделано.** `README.md:21` и `CLAUDE.md:81` приведены в соответствие с реальностью и с README:217/238: файлы закоммичены намеренно (секретов нет), локально при необходимости заполняются только `EMAIL_ACCOUNT`/`EMAIL_PASSWORD`.

### 1.2. `prod-smoke.yml` перезаписывал закоммиченный `.env.prod` из секрета **[ИСПРАВЛЕНО]**
- **Что не так.** Шаг «Write .env.prod» (`.github/workflows/prod-smoke.yml:21-24`) писал `.env.prod` из секрета `PROD_ENV_FILE` поверх файла из checkout. При этом закоммиченный `.env.prod` полный (проверено: `BASE_URL`, `STUDIO_URL`, `API_URL`, `PROD_TEST_*`, `USER_PASSWORD`, `USER_LOGIN_ADMIN`, `EMAIL_ACCOUNT`, `EMAIL_PASSWORD`), а шаг запуска (`:48-49`) уже передаёт `ENV_FILE=.env.prod` — файл из репозитория подхватывается сам.
- **Чем грозило.** Двойной источник правды: значения в секрете `PROD_ENV_FILE` могли молча разойтись с закоммиченным файлом, и прод-смоук гонялся бы против неверной конфигурации, которую не видно в репозитории.
- **Что сделано.** Шаг «Write .env.prod» удалён, YAML провалидирован (`yaml.safe_load` — OK).
- **Действие для пользователя.** Прогнать воркфлоу вручную один раз (Actions → Production Smoke Tests → Run workflow) и убедиться, что смоук зелёный на закоммиченном `.env.prod`. **После успешного ручного прогона секрет `PROD_ENV_FILE` можно удалить из настроек репозитория** — он больше нигде не используется.

### 1.3. Сломан автокоммит визуальных снапшотов в `aitv-visual-manual.yml`
- **Что не так.** Шаг автокоммита базовых снапшотов делает `git add "tests/visualSuite/**/__snapshots__"` (`.github/workflows/aitv-visual-manual.yml:117`). Такого пути в проекте нет **дважды**: (1) папка называется `tests/visual/`, а не `tests/visualSuite/`; (2) снапшоты Playwright лежат в `*.spec.ts-snapshots/`, а не в `__snapshots__/`. При `update_snapshots=true` `git add` не находит ничего, коммитить нечего — фича обновления базовых линий не работает.
- Дополнительно: `.github/workflows/aitv-visual-manual.yml:47` печатает `**Project:** ${{ inputs.project }}`, но во `workflow_dispatch.inputs` (`:5-22`) объявлены только `environment`, `update_snapshots`, `task` — входа `project` нет, строка всегда выводит пустое значение.
- **Чем грозит.** Молчаливо неработающий сценарий: инженер выбирает «Обновить базовые снапшоты», воркфлоу зелёный, но обновлённые PNG в репозиторий не попадают — визуальная регрессия остаётся на старых линиях. Плюс мусорная пустая строка в summary.
- **Что делать.** Исправить путь на реальный (`tests/visual/**/*-snapshots/**` или `git add tests/visual` целиком) и убедиться, что снапшоты, сгенерированные Docker-проектами, реально попадают под паттерн; убрать/добавить вход `project` в соответствии с тем, что печатается в summary. (В рамках текущей задачи только зафиксировано — правка не входила в scope.)

### 1.4. Противоречие инструкций по TC-конвенции между CLAUDE.md и агентами
- **Что не так.** `CLAUDE.md:191-204` («TC annotation convention») явно помечает `test('TC-ID: Title', ...)` как **WRONG** (ID должен быть только в `annotation`, не в имени теста). Но пайплайн-агенты требуют ровно обратного: `.claude/agents/code-reviewer.md:23` — `TC annotation format: `test('TC-ID: Title', ...)``; аналогичное правило в `.claude/agents/test-case-designer.md:38`.
- **Чем грозит.** Агенты `test-case-designer` и `code-reviewer` генерируют и ревьюят тесты по устаревшему правилу, прямо противоречащему главному источнику правды (CLAUDE.md). Новый код будет систематически нарушать конвенцию, а ревьюер этого не заметит — наоборот, будет требовать «неправильный» формат.
- **Что делать.** Синхронизировать `code-reviewer.md:23` и `test-case-designer.md:38` с CLAUDE.md: TC-ID только в `annotation: { type: 'TC', description: '...' }`, никогда в имени теста.

### 1.5. Битая команда prod-setup в CLAUDE.md **[ИСПРАВЛЕНО]**
- **Что не так.** `CLAUDE.md:64-67` содержал блок «Setup production accounts (run once)» с командой `ENV_FILE=.env.prod npx playwright test tests/production/setup.spec.ts --project=prodSmoke`. Файла `tests/production/setup.spec.ts` не существует (в `tests/production/` только `prodSmoke.spec.ts`), команда неисполнима.
- **Чем грозило.** Инженер, следующий инструкции при онбординге на прод, натыкается на «no tests found» и теряет доверие к документации.
- **Что сделано.** Блок удалён из `CLAUDE.md`.

## 2. Гонки и нарушение собственных правил в коде

Проект имеет чёткие «Coding Rules» в CLAUDE.md, но сам код местами их нарушает.

### 2.1. `waitForResponse` регистрируется ПОСЛЕ триггера (гонка)
Прямое нарушение правила CLAUDE.md «`waitForResponse` before trigger action».
- `src/pages/account/SecurityPage.ts:42-46` — сначала `await this.submitBtn.click()`, затем `await this.page.waitForResponse(...)`. Ответ `/api/account/email-2fa/set` может прийти до регистрации ожидания → флейки-падение по таймауту.
- `src/flows/AuthFlow.ts:122-126` (и аналогично около `:93`) — та же схема: клик, потом ожидание ответа.
- **Что делать.** Везде переставить: сначала `const p = page.waitForResponse(...)` с явным таймаутом, затем действие-триггер, затем `await p`. Добавить пример этого антипаттерна (со ссылкой на SecurityPage) в чек-лист `code-reviewer.md`.

### 2.2. `networkidle` на переходах
- `src/flows/AuthFlow.ts:131` использует `waitUntil: 'networkidle'` — запрещено правилом CLAUDE.md «Never use `waitUntil: 'networkidle'`» (особенно вредно рядом с плеером/стримингом, но и в общем случае нестабильно). Заменить на `domcontentloaded` + явное ожидание конкретного элемента/ответа.

### 2.3. Клики без предварительной проверки видимости/доступности
Нарушение правила «Check element availability before interacting».
- `src/pages/studio/StudioContentPage.ts:76-90` — все методы `clickXxxTab` кликают без `expect(...).toBeVisible()/toBeEnabled()`.
- `src/pages/account/SecurityPage.ts:33` и `:35` — клики по `setUpBtn`/`passwordInput` без проверок.
- `src/pages/channel/ChannelMainPage.ts:243` — клик без проверки.
- Шире: примерно **половина `expect` в Page Objects — без описательного сообщения** (порядка 343 из ~708), хотя CLAUDE.md требует информативное сообщение в каждом assert (`'... is not visible'`).
- **Что делать.** Добавить `toBeVisible()`+`toBeEnabled()` с сообщением перед каждым действием; постепенно проставить сообщения в «немых» expect. Это ровно тот класс, который `code-reviewer` должен ловить.

### 2.4. Инлайн-локаторы в методах (нарушение «locators — constructor only»)
Около 15 случаев, локатор создаётся прямо в методе, а не в конструкторе:
- `src/pages/channel/ChannelMainPage.ts` — строки 142, 147, 151-152, 158, 202, 228, 265.
- `src/pages/auth/LoginPage.ts` — строки 125, 147-148, 161.
- `src/pages/account/AccountPage.ts:122-123`, а также `UploadVideoPage.ts` и др.
- **Нюанс.** Часть из них — **параметризованные** локаторы (зависят от аргумента метода), которые физически нельзя вынести в конструктор. Это легитимное исключение из правила, но сейчас оно нигде не зафиксировано, из-за чего невозможно отличить настоящее нарушение от оправданного.
- **Что делать.** Вынести непараметризованные локаторы в конструктор; **явно узаконить в правилах исключение для параметризованных локаторов** (например: «локатор, зависящий от аргумента метода, допустимо строить внутри метода»).

### 2.5. Хардкод пароля вместо `process.env.USER_PASSWORD`
Пароль `'Admin1@@'` захардкожен в ~9 местах вместо использования `process.env.USER_PASSWORD` (который CLAUDE.md декларирует как единый источник):
- `src/flows/RegistrationFlow.ts:42, 54, 83, 108`
- `src/pages/account/SecurityPage.ts:36`
- `tests/.../handleValidationOnRegPage.spec.ts:25`
- Плюс `scripts/seedFixture.ts:64` — литерал `'1111'`.
- **Чем грозит.** Смена дефолтного пароля (или переменной окружения) потребует ручного поиска по всему коду; рассинхрон приведёт к массовым падениям аутентификации.
- **Что делать.** Заменить литералы на `process.env.USER_PASSWORD`; для сид-скрипта — вынести в константу/переменную окружения.

## 3. Флакинес-риски

### 3.1. Жёсткие `waitForTimeout`
28 вхождений `waitForTimeout` (23 в `src/`, 5 в спеках). Худшие:
- `src/flows/UploadVideoFlow.ts:196` — фиксированные 5 секунд.
- `src/utils/videoPlayerHelper.ts:33` и `:50`.
- `src/flows/AuthFlow.ts:117-121` — три паузы по 500 мс между вводом символов OTP.
- **Чем грозит.** Либо флейки (таймаут мал под нагрузкой стенда), либо замедление всего прогона (таймаут велик). **Что делать.** Заменять на ожидание конкретного состояния (`toBeVisible`, `waitForResponse`, `toHaveURL`); правило «без немотивированного `waitForTimeout`» стоит добавить в чек-лист ревью (см. §7).

### 3.2. Poll-loop с reload+sleep
- `src/pages/channel/ChannelMainPage.ts:227-234` — цикл `reload` + пауза 5 с, до 120 с суммарно. Дорого и нестабильно. **Что делать.** Заменить на `expect.poll`/`toPass` с ожиданием конкретного условия, либо на ожидание сетевого ответа.

### 3.3. Хрупкая serial-зависимость
- `uploadSeries.spec.ts` — SERIES-002 **тихо скипается**, если SERIES-001 упал (`describe.serial`). **Чем грозит.** Потеря покрытия маскируется под «skipped», в отчёте не виден реальный провал второго кейса. **Что делать.** Пересмотреть serial-цепочку или сделать зависимость явной (fail вместо тихого skip).

### 3.4. Ассерты, привязанные к длинным UI-строкам
- `src/pages/channel/ChannelMainPage.ts:147, 152, 267`, `src/flows/AuthFlow.ts:112` — сравнение с длинными текстами интерфейса. **Чем грозит.** Любая правка копирайта/локализации ломает тест. **Что делать.** Свести к устойчивым признакам (частичное совпадение по ключевому слову, `data-id`, роль), где полный текст не является предметом проверки.

## 7. Ревью инструкции по написанию тестов

Инструкция распределена между `CLAUDE.md` «Coding Rules», README «Правила написания тестов» и `.claude/agents/code-reviewer.md`.
